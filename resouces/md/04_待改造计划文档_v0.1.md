# 第二阶段：技术改造实施与进度报告 (v0.1)

## 1. 改造执行概述 (Execution Overview)

本阶段聚焦于系统核心组件的解耦与重构，涉及持久化层切换、认证体系收口、对象存储（OSS）真实接入及审计链路的完善。文档用于追踪各项改造任务的当前状态、技术细节及后续优化点。

### 1.1 执行优先级
1.  **持久化与媒体资产：** 数据层持久化（MySQL）与 OSS 参数化实现。
2.  **审计体系：** 审计日志双写（DB + Log）与 Outbox 模式落地。
3.  **认证与授权：** Sa-Token 鉴权闭环及 OAuth 2.0 真实链路对接。
4.  **容错与配额：** 限流策略升级及分布式配额管理。

---

## 2. 核心技术更新 (Key Technical Updates)

### 2.1 认证与网关入口 (Auth & Gateway)
- **API 收口：** 统一身份验证相关接口至 `/api/v1/auth/**` 路径。
- **网关策略：** 网关支持 `GUEST` 降级策略，对无效 Token 默认实施 401 拦截或降级访问。
- **安全加固：** 图形验证码增加失败上限逻辑，JWT 密钥实现配置驱动并支持冷启动告警。

### 2.2 韌性设计 (Resilience)
- **重试机制：** `UserQuotaClient` 接入统一重试组件，采用指数退避算法（Exponential Backoff with Jitter）。
- **异步审计：** Outbox 模式引入 `PROCESSING` 超时回收机制，确保审计事件的一致性。

## 3. 改造项完成情况记录 (Progress Tracking)

### 3.1 模块持久化改造 (Done)
- **全模块切换：** `user`, `content`, `media`, `ai` 模块均已完成由内存实现向 `ServiceImpl` + `MyBatis-Plus` 的迁移。
- **存储拓扑：** 对应核心库（`shizuki_user`, `shizuki_content` 等）已在各环境配置文件中生效。

### 3.2 OSS 媒体资产接入 (Done)
- **真实实现：** `AliyunOssClient` 替换 Mock 实现，支持分片上传、签名获取及删除。
- **配置抽象：** `OssProperties` 映射至 `common-config.yaml`，统一管理地域、桶名及配额。

### 3.3 审计日志入库 (Done)
- **双向同步：** 默认使用 `JdbcAuditLogService` 落库，失败自动降级至日志流。
- **检索能力：** 新增 `AuditLogQueryController` 支撑按 Trace ID 或用户维度进行审计回溯。

### 3.4 认证体系重构 (Done)
- **权限闭环：** `AuthEntryFilter` 实现网关级鉴权，支持身份透传及 ACL 权限点静态校验。
- **OAuth 闭环：** 支持 GitHub/LinuxDo 真实授权回调、账号自动绑定及 Token 生命周期管理。

## 4. 工程规范与交付标准 (Standards)

### 4.1 SQL 管理规范
- **文件分布：** `resouces/sql/` 存放各模块基线 SQL，文件名格式为 `shizuki_<module>.sql`。
- **变更追踪：** 每个脚本增量块必须标注 `modified_at` 及具体 `change` 用途说明。
- **运行时环境：** 生产环境变更强制通过 Flyway 迁移脚本执行。

### 4.2 配置管理标准
- **三层架构：** 
  1. `monolith.yaml`: 全局共用配置。
  2. `common-config.yaml`: 敏感与环境特定的私有配置。
  3. `common-config.example.yaml`: 离线配置基线模板。

## 2.6 问题 6：限流与配额升级（已完成）

### 已实现

- `RateLimitAspect` 键升级为用户/分组维度：
  - `rate-limit:<base>:u:<userId|guest>:g:<groups>`
- AI 配额改为数据库持久化计数：
  - `ai_quota_usage`
- AI 配额支持按用户分组动态解析：
  - ai-module 通过 `UserQuotaClient` 调 user-module 内部接口解析配额
  - user-module 通过 `group_quota_policy` 持久化配置返回结果
- 超限返回统一 ProblemDetail 结构，并携带 `remaining`：
  - 限流超限：`remaining=0`、`limit_key`、`window_seconds`
  - AI 配额耗尽：`remaining=0`、`quota_code`、`total`、`used`

### 仍待优化

- 限流当前为 Redis 优先 + 本地降级，下一轮建议补充分布式一致性压测与告警阈值。
