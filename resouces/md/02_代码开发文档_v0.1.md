# 个人学习陪伴网站｜代码开发文档 (v0.2)

## 1. 规范与约束 (Standards & Constrains)

### 1.1 API 设计规范
- **根路径：** `/api/v1`
- **URL 格式：** 采用 `kebab-case` 风格，使用复数名词表示资源（如 `/api/v1/posts`）。
- **字段命名：** JSON 响应及 Query 参数统一使用 `snake_case`。
- **错误模型：** 使用 `application/problem+json` 格式，包含 `code`, `message`, `trace_id`。

### 1.2 编码标准
- **工程模型：** 核心逻辑遵循 `Service` + `ServiceImpl` 模式。
- **依赖管理：** 强制使用构造器注入（Constructor Injection）。
- **代码规范：** 遵循阿里巴巴 Java 开发手册，启用 Checkstyle 静态检查。

---

## 2. 技术栈 (Tech Stack)

### 2.1 后端核心
- **框架：** Spring Boot 3.x
- **鉴权：** Sa-Token
- **持久层：** MyBatis-Plus
- **数据库：** MySQL 8.x
- **中间件：** Redis 7.x, Kafka

### 2.2 公共库职能 (Libs)
- `common-core`：定义基础响应模型、异常基类及全局错误码。
- `common-servlet`：集成 Trace 组件、审计切面、XSS 过滤器及动态限流器。
- `common-integration`：封装第三方服务（OSS 存储、OAuth2 授权）。

---

## 3. 工程结构 (Project Structure)

项目采用 Monorepo 结构进行管理：
```text
repo-root/
  apps/
    monolith-app/     # 主程序入口，负责路由分发与全局过滤
  modules/
    user-module/      # 用户中心、偏好同步、配额管理
    content-module/   # 博客、应用中心业务逻辑
    media-module/     # 媒体资产元数据管理与 OSS 交互
    ai-module/        # AI 会话流、模型对接与配额扣减
  libs/
    common-*          # 跨模块共享的二方库
```

---

## 4. 数据库设计要点 (Database Design)

1.  **命名规则：** 表名格式为 `<模块缩写>_<功能>`，如 `USR_ACCOUNT`。
2.  **通用字段：** 每张业务表必须包含 `id` (BIGINT), `create_time`, `update_time`。
3.  **约束命名：** 索引（IX）、外键（FK）及唯一约束（AK）需显式命名。

---

## 5. 开发建议顺序 (Development Order)

1.  搭建通用库 (`libs/*`) 与父工程依赖。
2.  实现 `monolith-app` 门户及 `user-module` 核心认证链路。
3.  同步开发 `content` 与 `media` 业务功能。
4.  集成 `ai-module` 对接大模型 API。
5.  执行端到端联调及性能压测。

> [!IMPORTANT]
> **代码提交规范：** 所有提交需包含类型前缀，如 `feat:`, `fix:`, `docs:`。禁止未经测试的代码合并至主分支。
