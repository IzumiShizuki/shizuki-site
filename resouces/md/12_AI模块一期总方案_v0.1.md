# 12_AI模块一期总方案_v0.1

- 文档日期：2026-03-24
- 适用仓库：`shizuki-site`
- 文档目标：冻结 AI 一期的产品边界、实现口径、评审结论与 beads 拆分，作为后续开发与审核的唯一基线

---

## 1. 需求分析

### 1.1 背景与目标

当前仓库已经具备以下 AI 基线：

1. 前端存在 `/ai-tavern` 页面占位与顶部 `AI Chat` 快聊按钮。
2. 后端 `ai-module` 已有会话、消息、角色卡、配额骨架。
3. 配额策略已落地：游客 `0` 轮，普通用户 `5` 轮，高权限组高额度。
4. 既有总体设计文档已经冻结了 `OpenAI Compatible + 角色卡 + 世界书 + 简化 RAG` 方向。

本次一期目标不是直接开发完整 Agent 生态，而是先把 AI 模块收敛为一个可执行的产品壳层，覆盖三块真实可落地能力：

1. `AI Hub / AI酒馆`：普通对话 + 酒馆模式。
2. `AI小镇`：RPGMaker 风格房间化场景，支持公开漫游与管理员特殊 NPC。
3. `自宅 companion`：管理员专属的个人陪伴 AI，占位未来虚拟伴侣演进路线。

`智能体 Agent 做事情的系统` 仅保留路线图与 beads backlog，不进入一期开发。

### 1.2 一期目标（冻结）

1. 把 `/ai-tavern` 升级为 AI 主入口，内部聚合 `快聊 / 普通对话 / 酒馆模式 / AI小镇 / 自宅 / Agent占位卡`。
2. 保留顶部 `AI Chat` 作为右侧快聊，不与完整 AI 路由冲突。
3. 酒馆模式支持角色卡、世界书、场景设定，普通对话保持轻上下文模式。
4. AI 小镇采用 RPGMaker 风格房间化展示，首批房间固定为 `图书馆 / 教室 / 自宅外部`。
5. 小镇支持 `RPGMaker 资源上传 + 解析 + 预览 + 配置挂接`，不做 RPGMaker 事件运行时。
6. 自宅 companion 提供人格配置、角色形象位、世界书绑定位、可选记忆模型开关。
7. 记忆模型只挂在 `特殊 NPC + 自宅 companion`，采用分层记忆与严格 `scope_id` 隔离。

### 1.3 一期不做（冻结）

1. OpenClaw / Agent 执行系统接入。
2. 多 Agent 协作、工具调用、工作流编排。
3. RPGMaker 完整事件解释器、碰撞系统、传送系统、脚本运行时。
4. 语音输入输出、Live2D 动作控制、桌宠常驻悬浮体。
5. 向量数据库级别的复杂 RAG 编排。
6. 面向普通用户开放自宅 companion 与记忆模型。

### 1.4 成功标准

1. 产品、前端、后端、测试使用同一份文档即可开始实现，不再补二次决策文档。
2. `/ai-tavern` 与顶部快聊入口职责清晰，用户不会在“快聊 / AI Hub / Apps”之间迷路。
3. AI 小镇与自宅的权限边界清晰：公开可逛、管理员可聊、非管理员不可进入自宅。
4. 后端接口与数据模型扩展方向明确，现有 `Map<String,Object>` 临时契约被替换为强类型 DTO 目标。
5. beads 任务树可直接被认领和执行，依赖顺序无二义性。

## 2. 产品调研

### 2.1 参考对象与可借鉴点

本次规划主要参考以下对象：

1. `SillyTavern`
   - 适合作为酒馆模式的交互参考。
   - 可借鉴能力：角色卡、世界书关联、场景化对话、普通聊天与角色聊天的差异化 prompt 装配。
2. `xmanrui/OpenClaw-bot-review`
   - 适合作为 AI 小镇的氛围与“像素风 Agent 空间”参考。
   - 可借鉴能力：像素化空间、房间感、角色具象化展示、配置驱动的空间观感。
   - 不借鉴点：本期不接入 OpenClaw 管理面、Gateway 健康、Bot 运行状态。
3. `moeru-ai/airi`
   - 适合作为自宅 companion 的长期演进方向参考。
   - 可借鉴能力：虚拟伴侣定位、人格容器概念、角色扩展槽、后续多模态演化空间。
   - 不借鉴点：本期不做语音、桌宠、VRM/Live2D 实时驱动。
4. 仓库现有 `ai记忆模型.md`
   - 提供了 AstrBot 风格分层记忆与 `scope` 隔离思路。
   - 本期只吸收分层记忆思想，不接 OpenClaw 执行器。

### 2.2 调研结论

#### 2.2.1 关于 AI Hub

1. 需要同时存在 `轻量快聊` 与 `完整 AI 页面` 两种入口。
2. 顶部按钮适合承担“快速发一句话”的场景，不适合承载角色卡、世界书、小镇漫游。
3. `/ai-tavern` 应升级为聚合页，而不是单一聊天页。

#### 2.2.2 关于酒馆模式

1. 酒馆模式必须和普通对话共用聊天壳层，否则实现与维护成本过高。
2. 两者真正差异不在组件本体，而在于：
   - prompt 装配
   - 角色绑定
   - 世界书触发
   - 场景设定注入
3. 一期可以继续沿用“关键词触发世界书”的简化路线，不强行上向量库。

#### 2.2.3 关于 AI 小镇

1. 如果直接做共享世界，会立刻引入多人状态同步与公共场景冲突问题，不适合一期。
2. 更合理的一期是：底层按个人实例建模，前台允许所有人公开浏览一套“公共投影”。
3. `图书馆 / 教室 / 自宅外部` 足以形成世界感，也足以支撑后续扩展更多房间。
4. RPGMaker 资源导入最稳妥的一期是“上传-解析-预览-挂接”，不是“运行 RPGMaker 游戏”。

#### 2.2.4 关于自宅 companion

1. 自宅需要独立于 AI 小镇特殊 NPC，因为它是长期陪伴入口，不是公共场景角色。
2. companion 一期必须先小范围内测，否则人格配置、记忆污染、权限风险都过高。
3. 管理员独占是合理冻结边界，后续可在权限策略里逐步开放。

#### 2.2.5 关于记忆模型

1. 若把记忆模型接到所有 AI，对一期复杂度与串味风险都太高。
2. 记忆模型应只用于“有持续人格需求的对象”，即特殊 NPC 与自宅 companion。
3. 记忆必须强制 `scope_id` 隔离，至少不能跨：
   - 用户
   - NPC
   - 自宅 companion
   - 私聊 / 公共场景

## 3. 需求撰写

### 3.1 信息架构

#### 3.1.1 AI 主入口

主入口固定为 `/ai-tavern`，命名保持不变，页面语义升级为 `AI Hub`。

内部固定六个区块：

1. `快聊`
   - 说明顶部 `AI Chat` 的定位
   - 展示最近会话与快捷入口
2. `普通对话`
   - 轻上下文对话
   - 不启用长期记忆
3. `酒馆模式`
   - 角色卡、世界书、场景设定
   - 支持结构化聊天配置
4. `AI小镇`
   - 公开漫游入口
   - 房间列表与地图预览
5. `自宅`
   - 管理员专属 companion
   - 后续虚拟伴侣演进入口
6. `Agent`
   - 仅做 roadmap 占位
   - 不提供可点击开发能力

#### 3.1.2 其他入口

1. 顶部 `AI Chat`
   - 保留为右侧快聊
   - 不直接跳转 AI Hub
2. 顶部菜单 `AI酒馆`
   - 进入 `/ai-tavern`
3. `Apps` 页面
   - 可保留 AI 小镇 / 自宅快捷卡
   - 但不是 AI 一期主入口

### 3.2 权限矩阵

| 能力 | 游客 | 登录用户 | ADMIN |
| --- | --- | --- | --- |
| 顶部快聊入口可见 | 是 | 是 | 是 |
| 快聊实际发言 | 否（0轮配额） | 是 | 是 |
| 进入 `/ai-tavern` | 是 | 是 | 是 |
| 普通对话 | 否（0轮配额） | 是 | 是 |
| 酒馆模式 | 否（0轮配额） | 是 | 是 |
| 查看 AI 小镇公开场景 | 是 | 是 | 是 |
| 与 AI 小镇特殊 NPC 对话 | 否 | 否 | 是 |
| 进入自宅 | 否 | 否 | 是 |
| 使用自宅 companion | 否 | 否 | 是 |
| 配置记忆模型 | 否 | 否 | 是 |
| 上传 RPGMaker 资源 | 否 | 否 | 是 |

补充规则：

1. 游客可见不代表可交互发言。
2. 非管理员在 AI 小镇中只能看到场景与 NPC 展示信息，不能发起特殊对话。
3. 自宅页面如果被非管理员直达，统一回退为权限提示页。

### 3.3 聊天模式需求

#### 3.3.1 快聊

1. 入口：顶部 `AI Chat`。
2. 目标：低心智负担快速发一句话。
3. 特征：
   - 不暴露世界书与角色管理 UI
   - 使用轻量上下文
   - 强调最近对话和剩余额度

#### 3.3.2 普通对话

1. 入口：`/ai-tavern` 内部标签。
2. 特征：
   - 仅使用当前会话上下文
   - 不加载长期记忆
   - 不强制角色设定
3. 适合场景：
   - 日常问答
   - 一次性任务
   - 轻陪伴

#### 3.3.3 酒馆模式

1. 必须支持：
   - 角色卡选择
   - 世界书绑定
   - 场景文本设定
   - 与普通模式共享聊天壳层
2. 角色卡来源：
   - 结构化 `character`
   - 导入型 `character_card_png`
3. 世界书采用关键词触发注入，不做复杂 RAG。

### 3.4 AI 小镇需求

#### 3.4.1 场景结构

一期固定三块房间：

1. `图书馆`
   - 偏资料、知识、设定浏览
   - 适合世界书氛围承载
2. `教室`
   - 偏教学、训练、问答感
   - 可作为后续教学 NPC 扩展点
3. `自宅外部`
   - 作为自宅入口场景
   - 负责引出管理员 companion

#### 3.4.2 能力边界

1. 支持公开浏览与房间切换。
2. 支持房间内 NPC 卡片与点位展示。
3. 支持管理员特殊 NPC 对话入口。
4. 支持上传 RPGMaker 资源并生成预览。
5. 不支持：
   - 玩家角色碰撞
   - 事件脚本运行
   - 传送逻辑脚本
   - 完整 RPGMaker 项目导入执行

### 3.5 自宅 companion 需求

一期 companion 必须具备：

1. 管理员专属访问。
2. 单独的人格配置。
3. 单独的角色形象位。
4. 单独的世界书绑定位。
5. 独立的记忆模型开关。
6. 为未来扩展保留：
   - 多角色皮肤
   - 陪伴日程
   - 情绪状态
   - 桌宠模式

### 3.6 记忆模型需求

分层固定为：

1. `profile`
   - 稳定身份、称呼、偏好、边界
2. `episodic`
   - 具体事件与对话片段
3. `summary`
   - 阶段性压缩摘要
4. `journal`
   - 按时间沉淀的日志型记录

固定规则：

1. 仅 `特殊 NPC + 自宅 companion` 可启用。
2. 每个记忆域必须带 `scope_id`。
3. `scope_id` 最少由 `user_id + actor_id + scene_type` 组成。
4. 关闭记忆时必须完全退化为普通上下文聊天。

## 4. 需求评审

### 4.1 评审结论（冻结）

以下决策经本轮评审后冻结，不再在一期实现中反复讨论：

1. `/ai-tavern` 保持为 AI 主路由，内部聚合多区块，不拆成多个一级路由。
2. 顶部 `AI Chat` 保留快聊职责，不改成跳完整页面。
3. AI 小镇按“个人实例底层 + 公开可逛前台”建模。
4. RPGMaker 范围冻结为“上传 + 解析 + 预览 + 配置挂接”。
5. 记忆模型只挂在“特殊 NPC + 自宅 companion”。
6. 自宅 companion 一期仅对管理员开放。
7. Agent 系统只建 roadmap 占位，不进入一期开发。

### 4.2 风险与处理策略

#### 4.2.1 入口过多导致认知混乱

处理：

1. 顶部按钮只承担快聊。
2. 完整能力统一收口到 `/ai-tavern`。
3. `Apps` 中的 AI 快捷卡仅作补充入口。

#### 4.2.2 AI 小镇范围失控

处理：

1. 房间数量冻结为 3。
2. RPGMaker 功能只做到预览层。
3. 不做运行时解释器。

#### 4.2.3 记忆串味与隐私污染

处理：

1. 只在管理员专用对象中启用。
2. 强制 `scope_id` 隔离。
3. 记忆关闭后完全不参与 prompt 注入。

#### 4.2.4 后端契约继续漂移

处理：

1. 从一期开始明确 `Map<String,Object>` 不是稳定契约。
2. 技术文档内直接列出目标 DTO。
3. 后续实现必须以强类型响应为准。

### 4.3 验收冻结项

以下内容在一期验收中必须被检查：

1. `/ai-tavern` 六区块入口完整可见。
2. 顶部 `AI Chat` 不与 `/ai-tavern` 冲突。
3. 酒馆模式可绑定角色卡与世界书。
4. AI 小镇非管理员不可与特殊 NPC 对话。
5. 非管理员不可进入自宅。
6. 记忆关闭时 companion 行为退化为普通聊天。
7. 记忆检索不得跨 `scope_id`。

## 5. 技术文档撰写

### 5.1 前端实现口径

#### 5.1.1 路由与页面壳层

保留现有：

1. 路由路径：`/ai-tavern`
2. 顶部菜单 key：`ai-tavern`
3. 顶部快聊按钮：`AI Chat`

升级方向：

1. `AiTavernPage.vue` 从单一聊天布局改为 AI Hub。
2. AI Hub 内部采用标签或分区切换承载六块能力。
3. 快聊侧栏继续沿用当前右侧面板形态，但 UI 文案对齐 AI Hub。

#### 5.1.2 组件拆分建议

建议新增组件分层：

1. `AiHubShell`
   - 统一页头、说明、权限提示、标签切换
2. `AiQuickChatPanel`
   - 快聊与最近会话摘要
3. `AiConversationWorkspace`
   - 普通对话 / 酒馆模式共享对话壳
4. `AiTavernConfigPanel`

5. `AiTownExplorer`

6. `AiCompanionHome`
   - 自宅 companion 对话与配置
7. `AiAgentRoadmapCard`
   - Agent 占位说明

### 5.2 后端接口口径

#### 5.2.1 会话聊天接口

1. `POST /api/v1/ai-sessions`
2. `GET /api/v1/ai-sessions`
3. `POST /api/v1/ai-sessions/{session_id}/messages`

3. `display_name`
4. `persona_json`
5. `avatar_asset_id`
6. `memory_enabled_flag`
7. `bound_worldbook_json`

记忆表通用字段：

1. `scope_id`
2. `owner_user_id`
3. `actor_code`
4. `content_text`
5. `importance_num`

### 5.4 记忆系统口径

#### 5.4.1 scope 规则

统一格式建议：

`<user_id>:<domain_type>:<actor_code>:<scene_code>`

示例：

1. `1001:town_npc:librarian:library`
2. `1001:companion:my_home_ai:home`

#### 5.4.2 注入顺序

仅当 `memory_enabled=true` 时按以下顺序组装：

1. `profile`
2. 最近有效 `summary`
3. 当前场景相关 `episodic`
4. 最近 `journal`
5. 本轮实时上下文

#### 5.4.3 防串味规则

1. 检索必须同时命中 `scope_id`。
2. 不允许跨 `actor_code` 复用记忆。
3. 小镇 NPC 和记忆 companion 绝不共享记忆域。
4. 公开场景浏览数据不写入记忆。

### 5.5 测试与质量门禁

后续实现阶段必须固定执行：

1. `mvn -DskipTests compile`
2. `modules/ai-module` 定向测试
3. 前端相关单测
4. `/ai-tavern` 与顶部菜单手工回归

固定验收场景：

1. `/ai-tavern` 六区块入口与权限提示正确。
2. 顶部 `AI Chat` 可继续快聊。
3. 普通对话与酒馆模式 prompt 装配不同。
4. 酒馆模式可绑定角色卡和世界书。
5. AI 小镇公开可逛，非管理员不可与特殊 NPC 对话。
6. 非管理员不可进入自宅。
7. RPGMaker 文件导入成功与失败路径都可预览反馈。
8. companion 支持人格配置与记忆开关。
9. 关闭记忆后行为退化为普通聊天。
10. 记忆检索不得跨 `scope_id`。
