# 01_需求分析

- 所属总览：[12_AI模块一期总方案_v0.1](../12_AI模块一期总方案_v0.1.md)

---

## 1. 背景与目标

当前仓库已经具备以下 AI 基线：

1. 前端存在 `/ai-tavern` 页面占位与顶部 `AI Chat` 快聊按钮。
2. 后端 `ai-module` 已有会话、消息、角色卡、配额骨架。
3. 配额策略已落地：游客 `0` 轮，普通用户 `5` 轮，高权限组高额度。
4. 既有总体设计文档已经冻结了 `OpenAI Compatible + 角色卡 + 世界书 + 简化 RAG` 方向。

本次一期目标不是直接开发完整 Agent 生态，而是先把 AI 模块收敛为一个可执行的产品壳层，覆盖三块真实可落地能力：

1. `AI Hub / AI酒馆`：普通对话 + 酒馆模式。
2. `AI小镇`：RPGMaker 风格房间化场景，支持公开漫游与管理员特殊 NPC。
3. `自宅 companion`：管理员专属的个人陪伴 AI，占位未来虚拟伴侣演进路线。

`智能体 Agent 做事情的系统` 仅保留路线图与 beads backlog，不进入一期开发。

## 2. 一期目标（冻结）

1. 把 `/ai-tavern` 升级为 AI 主入口，内部聚合 `快聊 / 普通对话 / 酒馆模式 / AI小镇 / 自宅 / Agent占位卡`。
2. 保留顶部 `AI Chat` 作为右侧快聊，不与完整 AI 路由冲突。
3. 酒馆模式支持角色卡、世界书、场景设定，普通对话保持轻上下文模式。
4. AI 小镇采用 RPGMaker 风格房间化展示，首批房间固定为 `图书馆 / 教室 / 自宅外部`。
5. 小镇支持 `RPGMaker 资源上传 + 解析 + 预览 + 配置挂接`，不做 RPGMaker 事件运行时。
6. 自宅 companion 提供人格配置、角色形象位、世界书绑定位、可选记忆模型开关。
7. 记忆模型只挂在 `特殊 NPC + 自宅 companion`，采用分层记忆与严格 `scope_id` 隔离。

## 3. 一期不做（冻结）

1. OpenClaw / Agent 执行系统接入。
2. 多 Agent 协作、工具调用、工作流编排。
3. RPGMaker 完整事件解释器、碰撞系统、传送系统、脚本运行时。
4. 语音输入输出、Live2D 动作控制、桌宠常驻悬浮体。
5. 向量数据库级别的复杂 RAG 编排。
6. 面向普通用户开放自宅 companion 与记忆模型。

## 4. 成功标准

1. 产品、前端、后端、测试使用同一份文档即可开始实现，不再补二次决策文档。
2. `/ai-tavern` 与顶部快聊入口职责清晰，用户不会在“快聊 / AI Hub / Apps”之间迷路。
3. AI 小镇与自宅的权限边界清晰：公开可逛、管理员可聊、非管理员不可进入自宅。
4. 后端接口与数据模型扩展方向明确，现有 `Map<String,Object>` 临时契约被替换为强类型 DTO 目标。
5. beads 任务树可直接被认领和执行，依赖顺序无二义性。
