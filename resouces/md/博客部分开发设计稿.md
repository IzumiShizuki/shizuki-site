你描述的这套「三栏阅读页 + 右侧可切换展开策略的文内目录 + 阅读进度环 + Markdown 下载 + Slidev/PPT 预览与自动生成」整体是可行的，而且结构上很“文档站（docs）化”，阅读体验会很好。

下面我按“你要的效果 → 推荐实现方式 → Slidev/PPT 的工程化落地 → 可进一步变得更精美的优化点”给你一套可以直接照着做的方案。

## 1) 你参考站点里“关键体验点”拆解成组件

### A. 三栏布局（左主导航 + 中正文 + 右 TOC）

* JavaGuide 这类页面的典型结构：左侧站点导航常驻，中间正文，右侧文内目录（或信息栏），标题下有阅读信息（字数 / 预计阅读时长）。JavaGuide 的示例里标题下就有“约 4858 字 / 大约 16 分钟”等信息。 ([JavaGuide][1])
* ArisuMika 的文章页标题下同样有“总字数 / 阅读时长”，并在侧边栏提供“目录”树。 ([ArisuMika'Blog][2])

你要做的基本就是：**一个文档站阅读页（docs reader）**，只不过你的内容源来自后端返回的 Markdown。

### B. 右侧 TOC 的两种模式

你要的模式其实可以定义为：

* **全部展开模式**：显示完整目录树（H2/H3/H4…）
* **当前小节展开模式**：只展开当前阅读到的 H2（或 H1）所在分支（含其子标题），其他分支折叠

这个体验和 Slidev 内置 `<Toc>` 组件的 `mode` 很像：它就支持 `all` / `onlyCurrentTree` / `onlySiblings` 三种展示策略。 ([cn.sli.dev][3])
（你可以在博客 TOC 组件里借用同样的命名，开发和用户理解都会更直观。）

### C. 返回顶部按钮 + 阅读进度环

右下角固定按钮，外圈环形进度跟随文章滚动进度变化。

### D. 右侧 TOC 顶部显示“行数 / 当前行 / 总行数”等

这里建议你不要展示“页面渲染后的行数”（会随屏幕宽度变化，意义不稳定），而是展示：

* Markdown **源文件行数**（稳定）
* 当前激活标题在 Markdown 的**起始行号**（稳定）
* 也可以加：当前已读百分比、已读字数、剩余阅读时间

要拿到“标题对应的 Markdown 行号”，推荐走 Markdown AST（语法树）解析，因为 AST 节点通常带位置信息（start/end 的 line/column）。unist 规范里明确了 Position/Point 的 start/end 概念与行列信息。 ([GitHub][4])

## 2) 推荐的整体技术架构（兼顾“好看”和“好维护”）

> 你没说你用 React 还是 Vue，所以我给两条都能落地的路线；你可以按熟悉度选。

### 路线 1：内容动态（后端返回 MD）→ 前端 SSR/CSR 渲染

适合：你文章存数据库、OSS、CMS，前端请求 API 获取 MD。

* **前端框架**：Next.js / Nuxt / Remix 任一都行
* **Markdown 处理**：统一用 `unified + remark + rehype`（Node/Edge 端做解析最舒服）
* **渲染策略**：

  * 后端/SSR：把 Markdown 转 HTML + 提取目录树 + 计算 meta（字数/行数/阅读时长），一次性返回给前端
  * 前端：只负责展示和 scrollspy（高亮当前标题）、模式切换

优点：首屏快、SEO 好、TOC/字数/行号这些都稳定。

### 路线 2：内容静态（仓库里就是 MD）→ 直接用“文档站框架”

适合：你文章就是 git 管理、静态站部署。

* Astro Starlight / Docusaurus / VitePress 这类**自带三栏布局 + TOC + 搜索**
* 你再加你想要的“切换目录展开策略 / 进度环 / MD 下载 / Slidev 集成”

优点：很多 UI 细节（排版、暗黑模式、移动端侧栏）开箱即用。
缺点：如果你“必须后端动态返回 MD”，那就不如路线 1 顺。

## 3) 三栏布局实现建议（可直接照抄的结构）

### 栅格布局（桌面端）

* 左侧主导航：`position: sticky`，top 为 header 高度
* 中间正文：最大宽度控制 + typography
* 右侧 TOC：`position: sticky`，跟随滚动；移动端变成抽屉/浮层

示意（伪代码）：

```tsx
<Header />

<div className="mx-auto max-w-[1400px] px-4">
  <div className="grid grid-cols-[260px_minmax(0,1fr)_240px] gap-6">
    <aside className="sticky top-[var(--header-h)] h-[calc(100vh-var(--header-h))] overflow-auto">
      <MainNav />
    </aside>

    <main className="min-w-0">
      <ArticleHeader meta={meta} />
      <ArticleContent html={html} />
    </main>

    <aside className="sticky top-[var(--header-h)] h-[calc(100vh-var(--header-h))]">
      <TocPanel toc={toc} meta={meta} />
    </aside>
  </div>
</div>

<ScrollToTopWithRing />
```

移动端建议：

* 左右侧栏隐藏
* 右侧 TOC 改成右下角按钮打开的 Drawer
* 主导航改成左上角汉堡菜单 Drawer

## 4) Markdown →（HTML + TOC + 行号 + 字数/阅读时长）一次性算出来

### A. 文章元信息（你想要的“标题下那一行”）

你参考的两个站点都有“字数 / 阅读时长”这类信息：JavaGuide 有“约 4858 字 / 大约 16 分钟”。 ([JavaGuide][1])
ArisuMika 有“总字数 / 阅读时长”。 ([ArisuMika'Blog][2])

建议你的 meta 至少包含：

* `wordCount`（字数/词数）
* `lineCount`（Markdown 行数）
* `readingMinutes`（预计阅读分钟）
* `updatedAt`（可选）
* `category/tags`
* `headings`（TOC 树 + 每个标题的 md 行号）

### B. 行号怎么来（稳定、可解释）

用 remark parse 出 mdast（Markdown AST），每个 heading 节点一般有 `position.start.line`。这个“位置”语义在 unist 的 Position/Point 里有定义（start/end 行列）。 ([GitHub][4])

### C. 阅读时长怎么估

中文博客建议按“汉字数”估算更贴近真实（比如 300~500 字/分钟可调），英文按 words/min。你也可以给用户一个“阅读速度设置”。

## 5) 右侧 TOC：两种展开模式 + scrollspy 高亮

### A. 模式定义（建议用 Slidev 同款命名）

* `mode = "all"`：展示全树
* `mode = "onlyCurrentTree"`：只展示“当前激活标题所在分支”
  （Slidev 的 `<Toc>` 就是这么设计的。 ([cn.sli.dev][3])）

### B. scrollspy（当前标题检测）

推荐用 `IntersectionObserver` 监听正文里的所有标题元素（h2/h3/h4），比 scroll 事件更省性能、更稳定。

实现思路：

1. 给渲染后的每个 heading 加 `id`
2. 观察它们进入视口的时机，取“最靠近顶部”的作为 activeId
3. activeId 改变时：

   * 高亮 TOC 项
   * 若 `onlyCurrentTree`：计算它所属的“顶层小节”（比如最近的 h2）并折叠其他分支

### C. TOC 顶部显示“行数/当前行等”

你可以在 TOC header 区显示：

* `总行数：{lineCount}`
* `当前：L{activeHeadingLine} / {lineCount}`
* `字数：{wordCount}，预计：{readingMinutes} 分钟`
* 再放两个按钮：`全部展开 / 当前小节`

## 6) 返回顶部 + 环形进度条（稳定方案）

进度 = `scrollTop / (scrollHeight - clientHeight)`
用 SVG circle 的 `stroke-dashoffset` 做环形进度最稳。

交互建议：

* 滚动超过 200~400px 才出现按钮
* 点击按钮：平滑滚动到顶部
* 进度环：跟随滚动实时更新
* 可加一个小提示：hover 显示“已读 42% / 还剩 3 分钟”

## 7) “下载 Markdown”的正确返回格式（后端 & 前端各给一份）

### A. 后端直接返回 MD 文件（最干净）

HTTP 响应建议：

* `Content-Type: text/markdown; charset=utf-8`
* `Content-Disposition: attachment; filename="xxx.md"`

示例（伪）：

```http
HTTP/1.1 200 OK
Content-Type: text/markdown; charset=utf-8
Content-Disposition: attachment; filename="resume-guide.md"

# 标题
...
```

### B. 前端拿到 md 字符串后生成下载（兼容性强）

```ts
function downloadMarkdown(filename: string, md: string) {
  const blob = new Blob([md], { type: 'text/markdown;charset=utf-8' })
  const url = URL.createObjectURL(blob)
  const a = document.createElement('a')
  a.href = url
  a.download = filename.endsWith('.md') ? filename : `${filename}.md`
  a.click()
  URL.revokeObjectURL(url)
}
```

## 8) PPT 预览 + Slidev 预览 + “一键生成 Slidev” 的工程化设计

你这个“PPT 直接打开 + Slidev 在线预览”的方向是合理的，尤其是你内容本身是 Markdown，Slidev 也是 Markdown 驱动，概念非常统一。Slidev 官方明确支持用 Markdown 分隔幻灯片（`---`）和 headmatter/frontmatter。 ([cn.sli.dev][5])

### A. 文章与演示的资源模型建议

给每篇文章附一个 `presentation` 字段（存在则展示预览入口）：

```yaml
---
title: xxx
date: 2026-03-03
presentation:
  pptUrl: ""          # 直接打开的 pptx 链接（可选）
  slidev:
    mdUrl: ""         # slidev markdown（可选）
    deckUrl: ""       # slidev build 后的静态站点（可选）
    export:
      pdfUrl: ""      # 可选
      pptxUrl: ""     # 可选
---
```

### B. Slidev 在线预览怎么做（推荐“build 后 iframe”）

Slidev 可以 `slidev build` 构建成可托管的静态 SPA。 ([cn.sli.dev][6])
你把 build 输出部署到你的站点某个路径，然后文章页里用 iframe 打开即可（注意 CSP 和同源）。

### C. 下载怎么做

* 如果你提供 Slidev deck（SPA），建议同时提供：

  * **PDF 下载**
  * **PPTX 下载**
* Slidev 官方支持 `slidev export` 导出 PDF / PPTX / PNG / Markdown，并说明 PPTX 导出是把每张幻灯片导出为图片（文本不可选择）。 ([cn.sli.dev][7])

> 这个点很关键：
> 如果你期望“生成的 pptx 还能在 PowerPoint 里继续编辑文字/排版”，那 Slidev 导出的 PPTX **不太满足**（因为是图片化）。 ([cn.sli.dev][7])
> 但如果你只是要“可下载、可播放、样式一致”，Slidev 的 PPTX 导出完全 OK。

### D. “没有 slidevMd/pptUrl 的文章 → 一键生成 slidev”

我建议你把它做成一个**后端任务**（避免在浏览器里做 heavy work），流程：

1. 后端拿到文章 Markdown
2. 解析 headings + 列表（用于拆分 slide）
3. 生成 slidev markdown（可编辑、可保存）
4. 可选：触发 `slidev build` 生成 deckUrl
5. 可选：触发 `slidev export --format pdf/pptx` 生成下载文件 ([cn.sli.dev][7])

#### 生成 Slidev Markdown 的“最低可用模板”

* 第 1 页封面（大标题）
* 第 2 页目录（用 Slidev 内置 `<Toc>`，并设置深度/模式）
* 后面每个 H2 做一页；H3/H4 或该段落的小点拆成若干页（可控制最大要点数，避免一页塞爆）

Slidev 的 `<Toc>` 用法在官方内置组件文档里有说明。 ([cn.sli.dev][3])

一个示例（生成出来的 slidev md 大概长这样）：

```md
---
title: 你的文章标题
theme: default
---

# 你的文章标题
副标题 / 作者 / 日期

---

# 目录
<Toc minDepth="2" maxDepth="3" mode="all" />

---

# 第一部分（来自 H2）
- 要点 1
- 要点 2
- 要点 3

---

# 第一部分 - 要点 1（如果要拆更细）
这里放解释、图、代码等

---
# 第二部分（来自下一个 H2）
...
```

你要的“全部展开 / 当前小节展开”也可以直接体现在 Slidev TOC 页里：

* `mode="all"` vs `mode="onlyCurrentTree"`（和你博客右侧 TOC 的模式概念完全一致）。 ([cn.sli.dev][3])

## 9) 你这套选型的评价 + 可以更精美/更优化的方向

### 你当前的选型优点

1. **一致性强**：文章是 Markdown，演示也是 Markdown（Slidev），内容生产成本很低。 ([cn.sli.dev][5])
2. **可托管、可分享**：Slidev 可以构建成静态 SPA，适合嵌入到博客里做预览。 ([cn.sli.dev][6])
3. **导出能力完整**：官方支持导出 PDF/PPTX/PNG/MD。 ([cn.sli.dev][7])

### 你需要提前接受的取舍

* **Slidev 的 PPTX 导出本质是图片化**：更像“可播放交付物”，不是“可二次编辑的源文件”。 ([cn.sli.dev][7])
* **运行时生成 build/export 有成本**：建议做缓存、做队列、做“首次生成后复用”的策略。

### 让它“更精美”的 6 个升级点（强烈建议）

1. **排版系统**：用 typography（例如 prose 风格）+ 自定义标题间距、列表间距、代码块字体；细节决定质感
2. **代码块体验**：Shiki 高亮 + 复制按钮 + 行高亮 + 折叠长代码（Slidev 里也强调了代码展示能力）。 ([cn.sli.dev][5])
3. **阅读模式**：像 JavaGuide 页面底部有“沉浸阅读”入口那样，提供一键隐藏左右侧栏。 ([JavaGuide][1])
4. **TOC 更“信息化”**：右侧 TOC header 显示：已读字数 / 剩余分钟 / 当前标题行号（从 AST 来）
5. **动效克制但高级**：目录展开折叠、active 高亮、返回顶部按钮出现/消失，用轻量 motion（不要花哨）
6. **生成 Slidev 后允许“二次编辑”**：自动生成只能做到 60~80 分好看；你提供一个在线编辑入口（哪怕是最简单的 md 编辑器）会让最终效果质变

### 如果你追求“可编辑 PPTX”（非图片化）的替代路线

你可以保留 Slidev 作为“在线演示/分享版”，同时提供另一条“真 PPTX”的生成：

* 后端用 PptxGenJS 之类生成可编辑 pptx（更像 PowerPoint 原生对象）
  缺点是：你要设计一套“Markdown → PPT 排版”的规则，会比 Slidev 更费精力（但输出更“办公友好”）。

## 最后给你一个“最小可落地的开发顺序”

1. 先把 **文章页三栏布局 + Markdown 渲染**做出来
2. 把 **TOC 树（含行号）+ scrollspy + 两种展开模式**做出来
3. 加上 **标题下 meta（字数/阅读时长/行数）** + 右下角 **进度环返回顶部**
4. 做 **Markdown 下载**
5. 再做 **presentation 模块**：

   * 有 pptUrl → “打开/下载”
   * 有 slidev deckUrl → iframe 预览 + 下载
   * 都没有 → “一键生成 Slidev”（后端生成 + 缓存）

如果你愿意，我也可以按你更倾向的技术栈（Next.js / Nuxt / 纯前端 CSR）把 **TOC 模式切换 + 行号显示 + 进度环按钮**这三块给你写成一套可直接复制的组件代码结构（包括状态管理和样式组织方式）。
