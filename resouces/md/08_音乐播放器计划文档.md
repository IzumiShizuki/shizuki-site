# 音乐播放器后端实施方案 (v0.1)

## 1. 方案目标 (Objectives)

本方案定义了音乐模块后端的实施边界，聚焦于多渠道音乐资源获取（以 Spotify 为主）、配额控制管理以及内部服务间的安全通信机制。本阶段不涉及前端播放器业务逻辑的重构。

### 1.1 核心建设边界
- **Provider 集成：** 首期支持 Spotify 作为第三方数据源，仅限搜索与试听（Preview）功能。
- **凭据管理：** 支持用户级别 Spotify Token 绑定，但不涉及 OAuth 回流登录逻辑。
- **配额体系：** 建立基于分组策略（Group Policy）的选歌与上传容量管理。
- **内部安全：** 对服务间敏感数据传输接口实施 HMAC 签名验证。

---

## 2. 业务规则与模型 (Business Rules)

### 2.1 配额与权限策略
- **游客权限：** 仅限访问系统预设的默认歌单。
- **选歌配额：** 普通登录用户具备固定次数的“选歌”额度（`music_song_pick_total`），耗尽后需绑定自有 Provider 凭据。
- **存储配额：** 上传曲目容量受分组配额控制（`music_upload_bytes_total`），文件统一托管于 OSS。
- **缓存机制：** 高等级用户（Friend 及以上）触发的第三方曲目请求支持 OSS 缓存，以优化 API 消耗。

### 2.2 数据模型扩展
- **USR_PROVIDER_SECRET:** 存储加密的用户级 API Keys。
- **MDA_MUSIC_PLAYLIST:** 系统及个人歌单管理。
- **MDA_MUSIC_PICK_USAGE:** 记录用户选歌消耗流水。
- **MDA_ASSET:** 资源基表扩展 `AUDIO` (code=4) 类型支持。

---

## 3. 接口规范 (API Specifications)

### 3.1 消费端接口 (Client)
- **`POST /api/v1/music/picks`**: 执行选歌操作，触发配额核销。
- **`GET /api/v1/music/spotify/search`**: 基于 Spotify 的曲目搜索。
- **`PUT /api/v1/me/music/api-keys/{provider}`**: 用户自助绑定第三方插件凭据。

### 3.2 管理端接口 (Admin)
- **`GET /api/v1/admin/group-quota-policies`**: 管理全站分组配额策略（含 AI、音乐）。
- **`PUT /api/v1/admin/music/default-playlist`**: 维护系统默认背景音。

### 3.3 内部安全接口 (Internal)
- **`GET /api/v1/internal/music/api-keys/**`**: 用于各模块间获取明文凭据。
- **安全校验：** 强制实施 `HMAC-SHA256` 签名及 Nonce 防重放校验，确保敏感信息仅在可信域内流动。

---

## 4. Spotify 集成细节 (Spotify Integration)

- **Token 降级链：** 请求时优先检索用户级 Token；若缺失，则回退至系统级 `client_credentials` 凭据。
- **异常处理：** 定义标准的 `MUSIC_SPOTIFY_UPSTREAM_ERROR` 等错误码。当 `preview_url` 缺失时，接口返回空字符串并标记状态位，不抛出硬阻断异常。

---

## 5. 验收与交付标准 (Acceptance)

- [x] **配额约束生效：** 无 Key 状态下的第 6 次选歌请求应被准确截断。
- [x] **签名验证闭环：** 内部接口在无签名或签名偏移 300s 以上时返回 403。
- [x] **搜索链路：** Spotify 搜索功能及预览链接解析需符合预期指标。

> **配置提醒：** 请查阅 `resouces/yaml/monolith.yaml` 确保 `shizuki.internal.auth.shared-secret` 等核心安全配置已正确初始化。
