# 基础设施建设方案：框架与工具类搭建 (v0.1)

## 1. 方案概述 (Overview)

本阶段旨在完成系统的核心工程底座，包括单体模块化结构的组织、通用二方库（Library）的封装、基础中间件的编排以及全局性的横切关注点（鉴权、限流、追踪）的落地。该阶段不涉及具体的业务逻辑开发。

---

## 2. 建设目标与验收标准 (Goals & DoD)

### 2.1 建设目标
1.  **工程骨架：** 完成单体模块化工程搭建，支持各业务模块以隔离方式运行在同一应用实例内。
2.  **配置中心化：** 实现基于 `monolith.yaml` 与 `common-config` 的统一配置分发。
3.  **核心门控：** 在入口层实现统一鉴权、分布式限流及全链路追踪（Trace）。
4.  **基石库落地：** 交付 `common-core`、`common-servlet`、`common-integration` 三个核心通用包。

### 2.2 验收标准 (DoD)
- 服务整体健康检查状态正常 (`/actuator/health`)。
- 鉴权逻辑闭环：受保护接口在未授权状态下返回 401/403。
- 限流策略生效：请求量超过定义阈值时返回标准的限流错误码。
- 全链路追踪：单次请求的日志可通过 `trace_id` 实现完整串联。

---

## 3. 核心通用库职能定义 (Common Libraries)

### 3.1 `common-core`：通用领域模型
定义系统的“通用语言”，确保模块间通信的一致性。

| 组件 | 职能描述 | 状态 |
| :--- | :--- | :--- |
| **ApiResponse** | 统一 REST 响应结构 (`code`, `message`, `data`) | 已实现 |
| **ErrorCode** | 全局错误码枚举，规范业务异常语义 | 已实现 |
| **BusinessException**| 业务异常基类，支持与全局异常处理器联动 | 已实现 |
| **RetrySpec** | 统一重试策略（包含指数退避与抖动算法） | 已实现 |

### 3.2 `common-servlet`：Web 增强套件
封装 Web 层的横切逻辑，降低业务模块重复开发成本。

| 组件 | 职能描述 | 状态 |
| :--- | :--- | :--- |
| **GlobalExceptionHandler** | 异常翻译器，统一输出 `problem+json` 响应 | 已实现 |
| **TraceIdFilter** | 负责 `trace_id` 的生成与下游链路透传 | 已实现 |
| **RateLimitAspect** | 基于 Redis 的高性能动态限流切面 | 已实现 |
| **AuditLogAspect** | 自动采集关键操作的审计日志 | 已实现 |
| **HtmlSanitizer** | 提供基于策略的 HTML 内容净化服务（防 XSS） | 已实现 |

### 3.3 `common-integration`：外部适配层
管理与第三方服务（存储、OAuth）的交互逻辑。

| 组件 | 职能描述 | 状态 |
| :--- | :--- | :--- |
| **ObjectStorageClient** | 对象存储（OSS）的统一抽象接口 | 已实现 |
| **AliyunOssClient** | 阿里云 OSS 的具体适配实现 | 已实现 |
| **OAuthStateService** | 负责 OAuth 授权过程中 `state` 参数的安全性校验 | 已实现 |
| **OAuthProviderFactory**| 多平台（GitHub/LinuxDo）登录策略工厂 | 已实现 |

---

## 4. 实施路线图 (Roadmap)

1.  **工程奠基：** 建立 Maven 父工程与各模块 POM 关系。
2.  **基础设施编排：** 编写 `compose.yaml`，拉起 MySQL、Redis 与 Kafka 容器环境。
3.  **门户搭建：** 实现 `monolith-app` 入口层，集成鉴权与审计拦截器。
4.  **模块接入：** 将 `user`、`content` 等业务模块接入公共库。
5.  **链路验收：** 通过端到端冒烟测试确认工程骨架的稳健性。

> [!NOTE]
> 第一阶段完成后，系统将具备承接正式业务代码的条件。所有后续开发的模块必须强制遵循本阶段定义的各项规范。
