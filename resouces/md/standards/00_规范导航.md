# 项目规范导航

> 本目录是当前仓库的规范总入口，用来收口命名、术语、组件分层、前端状态结构，以及前后端 API / SQL / 模型命名边界。

## 1. 使用原则

发生规范冲突时，按以下优先级处理：

1. `resouces/md/standards/*` 中的专门规范
2. 可执行或可校验的权威来源
   - `scripts/check_sql_conventions.sh`
   - `scripts/check_backend_contracts.sh`
   - `resouces/sql/00_object_name_mapping.yaml`
3. 现有总设计与开发规范文档
   - `resouces/md/总体设计文档_f.md`
   - `resouces/md/02_代码开发文档_v0.1.md`
   - `resouces/md/05_AI协作开发规范_v0.1.md`
   - `resouces/md/11_前端界面审查与组件整合路线_v0.1.md`

说明：

- 本目录优先解决“怎么叫、放哪层、字段怎么命名、哪些是遗留兼容”。
- SQL 对象名、Flyway 命名、字段注释等门禁，以脚本和映射文件为最终准绳。
- 顶层目录 `fronted`、`resouces` 当前视为 `legacy path`，本轮不重命名，但后续新增内容不得继续制造同类拼写漂移。

## 4. 本轮冻结结论

- `轻应用` 是用户可打开的工具型功能，代码前缀统一 `LightApp*`。
- `Widget / 小组件` 仅保留给首页嵌入块，不再与 `轻应用` 混用。
- 悬浮入口系统统一叫 `Floating Dock / 悬浮栏`。
- 悬浮栏圆形触发器统一叫 `Floating Ball / 悬浮球`。
- “menu 下那个框”统一叫 `Dock Panel / 扩展面板`，不再把 `picker-panel` / `submenu-panel` 作为正式术语。
- 轻应用持久化主结构固定为 `rail_slots`，`ball_slots` 只作为历史兼容输出与输入别名。
- 后端 Controller 对外稳定契约统一使用强类型 `*Request` / `*Response` / `*Summary`，不再把 `Map<String, String>` 状态响应视为正式契约。
- `model/request|response` 下的 `.dto` 物理包路径暂时冻结为 legacy package；第一阶段统一类名和接口签名，不做物理迁包。
