# 命名规范

## 1. 基本约束

- HTTP path：`kebab-case`
- Query / JSON / 持久化字段：`snake_case`
- 前端运行时变量、props、composables：`camelCase`
- Java 类型、Vue 组件名：`PascalCase`
- CSS class：优先语义化短横线命名

## 2. 标识命名

- 稳定业务标识统一使用 `code`
- 主键或唯一对象标识统一使用 `id`
- 判别字段优先使用 `kind`
- `type` 仅用于历史兼容、外部协议或枚举名已固定的场景

推荐：

- `post_code`
- `group_code`
- `collection_id`
- `item_kind`
- `item_ref`

避免：

- `key` 作为稳定业务主标识
- `flag` 承载多值语义
- `type` 同时表示“种类 + 业务模式 + UI 状态”

## 3. 状态命名

布尔状态统一使用清晰前缀：

- `active*`
- `visible*`
- `expanded*`
- `collapsed*`
- `loading*`
- `saving*`
- `selected*`

运行时枚举状态优先使用：

- `*Kind`
- `*Mode`
- `*Status`

本仓库冻结示例：

- `panelKind` 优于 `panelMode`，因为这里表达的是当前扩展面板种类
- `activeCollectionId` 优于 `panelCollectionId`

## 4. 组件命名

- 路由级：`*Page`
- 页面内主视图：`*View`
- 可拖拽轻应用：`*Window`
- 模态弹层：`*Dialog`
- 移动端底部层：`*Sheet`
- 锚点短暂浮层：`*Popover`
- 导航：`*Nav` / `*Rail` / `*Sidebar` / `*Dock`
- 布局：`*Layout` / `*Shell` / `*Stage`
- 内容块：`*Card` / `*Section` / `*Item`

禁止继续新增：

- 纯语义不清的 `Wrapper`、`Box`、`Block`
- 路由级组件却使用 `*View`
- 轻应用窗体却使用 `*Panel`

## 5. 模型命名

- 入参：`*Request`
- 出参：`*Response`
- 列表简版：`*Summary`
- 选项集：`*OptionResponse`
- 持久化实体：`*Entity`

允许保留但需登记：

- 已存在且已被多处引用的 `*Dto`
- 已存在且属于临时过渡期的 `*View`

## 6. 反模式

- 同一对象同时出现 `collection_id`、`collectionId`、`folderId` 三套说法
- 同一 UI 结构同时出现 `panel`、`submenu`、`picker` 三套词
- 组件文件名表达布局语义，状态变量却表达业务语义，互相对不上
